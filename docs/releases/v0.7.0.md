# v0.7.0 - Quality Gate Baselines

Date: 2026-06-26

Release status: ready

Release positioning: offline report validation and offline simulation validation
quality gates.

## Scope

v0.7.0 was built as a set of small, reviewable quality-gate PRs on top of
v0.6.0.

## PR 1: Report Validation and Evidence Index Baseline

The first v0.7.0 PR strengthened `validate-report` as an offline report evidence
chain quality gate.

Added capabilities:

- `validate-report --report-dir <dir>`
- explicit file path validation mode
- deterministic `evidence_index.json`
- structured `report_validation.json`
- validation findings for missing evidence refs, broken evidence refs, missing
  report sections, and missing audit coverage

## PR 2: Simulation Scenario Matrix Baseline

The second v0.7.0 PR adds a deterministic offline/mock simulation scenario
matrix for `validate-simulation`.

Added files:

- `docs/simulation_scenario_matrix.md`
- `data/sample_simulation/scenario_matrix.json`
- unit and integration tests for matrix-driven simulation validation

Covered matrix scenarios:

| Scenario | Expected result |
| --- | --- |
| nominal flight | `PASS` |
| battery sag / low battery degradation | `FAIL` |
| GPS degradation | `FAIL` |
| motor vibration anomaly | `FAIL` |
| severe temperature issue | `FAIL` |
| missing validation constraint | `REVIEW_REQUIRED` |
| missing telemetry fields | `INVALID_INPUT` |
| inconsistent simulation metadata | `INVALID_INPUT` |

`INVALID_INPUT` is a matrix expectation for payloads that fail before
`SimulationRun` creation; it is not a new `SimulationRun.status`.

Determinism improvements:

- `simulation_run.json` now uses deterministic `id` and timestamp values for the
  same scenario/result pair.
- simulation validation evidence refs are sorted by rule id, field, and source.
- validation audit `rules_triggered` remains sorted.

## PR 3: Validation Quality Gate and Release Readiness

The third v0.7.0 PR closes the quality-gate release path by documenting a
repeatable local readiness checklist for report validation, evidence index
generation, and offline simulation validation.

Added documentation:

- `docs/v0.7.0_release_readiness.md`
- README release readiness entry
- Codex workflow guidance for v0.7.0 release checks

Readiness gates:

- full `pytest`
- temporary `run-mvp` report generation
- `validate-report --report-dir <dir> --write-index`
- generated `evidence_index.json`
- generated `report_validation.json`
- `validate-simulation` with the mock/offline fixture
- focused simulation matrix tests
- GitHub Actions success on Python 3.11 and Python 3.12
- offline-only and advisory-only safety boundary confirmation

## PR 4: Simulation Operational Rules Baseline

The fourth v0.7.0 PR expands `validate-simulation` from a scenario matrix
baseline into a more explicit offline operational rules baseline.

Added capabilities:

- structured `rule_results` in `simulation_run.json`
- return-home altitude validation
- low-battery return strategy validation
- communication link loss duration validation
- geofence margin risk validation
- wind disturbance mission completion validation
- payload/endurance margin validation
- rule-level `evidence_refs`
- rule-level `human_review_required=true`

Additional matrix scenarios:

| Scenario | Expected result |
| --- | --- |
| return-home altitude breach | `FAIL` |
| low-battery return not triggered | `FAIL` |
| communication link loss | `FAIL` |
| geofence margin risk | `FAIL` |
| wind disturbance with low mission completion | `FAIL` |
| payload/endurance margin breach | `FAIL` |

These cases validate imported offline/mock result fields only. They do not
start PX4, ArduPilot, Gazebo, SITL, or any external simulator, and they do not
execute MAVLink commands or real flight-control actions.

## Safety Boundary

v0.7.0 remains an offline advisory agent release.

It does not include:

- real drone connection
- MAVLink command execution
- arm/disarm
- takeoff/landing/RTL
- mission execution
- motor start
- firmware upload
- flight-controller parameter writing
- real PX4, ArduPilot, Gazebo, SITL, or external simulator startup or connection

All simulation validation remains offline-only, mock-import-only, and
advisory-only. Safety and maintenance conclusions continue to require human
review.

## Current Limitations

- The simulation matrix is a deterministic mock/offline import baseline, not a
  real simulator runner.
- `INVALID_INPUT` matrix cases document rejected payloads and do not produce a
  `SimulationRun`.
- Audit run ids and creation timestamps still reflect the local skill execution
  event, even though rule ordering and output payloads are deterministic.

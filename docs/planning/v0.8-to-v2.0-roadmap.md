# v0.8.0 到 v2.0.0 路线图

当前项目已发布到 `v0.9.0 - Work Order Drafts`。后续路线建议继续坚持：

数据可信度 -> 仿真验证 -> 报告闭环 -> 工单闭环 -> 稳定版 -> 机队与平台化能力。

## 已完成基线

### v0.6.0：真实样例日志验证

`v0.6.0` 已完成 parser coverage 文档、sample fixture policy、PX4 / ArduPilot placeholder 目录、mock fixture 与 future sanitized fixture 的边界说明，以及真实样例缺失时的验证测试。

当前策略是不提交未经确认的真实 `.ulg` / `.bin` 二进制样例。仓库保留 deterministic mock fixtures 和真实/脱敏样例 placeholder 目录。

### v0.7.0：质量门禁基线

`v0.7.0` 已完成：

- `validate-report` 报告证据链验证。
- `evidence_index.json` 和 `report_validation.json`。
- 离线/mock simulation scenario matrix。
- `validate-simulation` operational rules。
- `simulation_run.json` 中的结构化 `rule_results`。
- rule-level `evidence_refs` 和 `human_review_required=true`。
- v0.7.0 release readiness checklist。

## v0.8.0：报告闭环增强

目标：让报告真正成为“单次运维复盘材料”，完整包含日志、异常、诊断、维护、仿真和审计信息。

建议拆成小 PR：

1. 报告纳入仿真结果。
   - `generate-report --simulation <simulation_run.json>`
   - `ops_report.md` 增加仿真验证章节。
   - 报告展示仿真状态、场景 ID、规则结果、证据引用和人工复核要求。

2. 报告纳入审计与 parser metadata。
   - 展示 audit summary。
   - 展示 parser format / parser metadata。
   - 展示 human review checklist。

3. PDF 同步增强。
   - 确认 `generate-report --pdf` 不回归。
   - 确认新增 Markdown 章节在 PDF 中可读。
   - 继续验证中文字体和 Poppler 渲染。

验收标准：

- 不提供 `--simulation` 时，现有报告行为不变。
- 提供 `--simulation` 时，Markdown 报告包含仿真验证章节。
- 报告中的仿真结论可追溯到 `simulation_run.json` 的 `evidence_refs`。
- PDF 输出不回归。
- 仍保持 offline-only、mock-import-only、advisory-only。

## v0.9.0：工单草稿闭环

目标：把维护建议升级为可审计、待人工确认的工单草稿。

建议能力：

- 新增 `WorkOrderDraft` schema。
- 将 `MaintenanceRecommendation` 转换为工单草稿。
- 新增 CLI：`drone-ops generate-work-orders`。
- 工单字段包含 asset id、component、priority、action、reason、evidence_refs、required_approval、estimated_effort、reviewer、status。
- 支持导出 JSON / Markdown。
- audit 记录工单草稿生成过程。

安全边界：

- 不调用真实 CMMS、Jira、飞书或企业微信 API。
- 不自动派单。
- 不自动执行维护动作。
- 工单只是草稿，必须人工确认。

## v1.0.0：稳定离线运维 Agent

目标：冻结并稳定第一个离线主版本，把 `v0.1.0` 到 `v0.9.0` 已形成的能力收成可靠的 offline advisory baseline，不再塞入大功能。

重点：

- CLI contract freeze。
- 主要 schema 稳定化。
- 所有核心 CLI 都有测试。
- 所有 skill 都有清晰的 `SKILL.md`、schema、examples 或测试说明。
- audit / evidence_refs 贯穿主流程。
- README、docs、release notes 完整。
- GitHub Actions 全绿。
- 安全边界回归测试完整。

## v1.1.0：机队健康趋势分析

目标：从单次飞行扩展到多架无人机、多次任务的趋势分析。

建议能力：

- `FleetAsset` / `FleetHealthSummary` schema。
- 多个 flight summary 聚合。
- 电池健康趋势、振动趋势、GPS 质量趋势、链路质量趋势。
- 组件异常频率统计。
- 机队风险排名。
- CLI：`drone-ops fleet-summary`、`drone-ops compare-flights`。
- 输出 fleet report，并能追溯到单次飞行证据。

## v1.2.0：本地 Web Dashboard MVP

目标：从 CLI 扩展到本地只读可视化看板。

建议能力：

- 本地 FastAPI backend。
- 简单 Web UI。
- 上传或选择本地样例日志。
- 查看分析结果、报告、audit、evidence refs 和仿真验证结果。

边界：

- 不做远程部署。
- 不做多用户权限。
- 不做实时飞控连接。
- 不控制真实无人机。

## v1.3.0：规则包和 skill 版本管理

目标：让系统更像可复用、可审计的规则和 skill 平台。

建议能力：

- `RulePack` schema。
- 规则版本号。
- skill version registry。
- audit 记录 rule pack version。
- 支持不同机型、任务类型的规则配置。
- CLI：`drone-ops list-skills`、`drone-ops list-rule-packs`、`drone-ops validate-rule-pack`。

## v1.4.0：诊断与报告评测

目标：让诊断和报告不只是“能输出”，而是能评估质量。

建议能力：

- 正式化 `evals/`。
- golden diagnosis cases。
- expected hypotheses 和 expected maintenance recommendations。
- 评分指标：anomaly recall、hypothesis ranking quality、evidence completeness、safety flag correctness、report completeness。
- CLI：`drone-ops run-evals`。
- 生成 eval report。

## v1.5.x：平台化准备

目标：为 v2.0.0 平台化提前补齐工程边界。

建议能力：

- 本地 project/workspace 格式。
- report bundle import/export。
- reviewer / approval model。
- 数据保留和脱敏策略。
- offline adapter contracts。
- 安全和隐私审查清单。

## v2.0.0：组织级运维平台基线

目标：从“工具”升级为“组织级运维决策平台基线”。

建议包含：

- CLI + 本地 Web dashboard。
- 多资产管理。
- 多日志格式。
- 多任务报告。
- 机队健康趋势。
- 工单草稿或 mock CMMS。
- 规则包和 skill 版本管理。
- evals。
- 审计闭环。
- reviewer / approval 概念。
- 可导出的组织级月报 / 周报。

关键边界：

即使到 `v2.0.0`，系统也不应直接控制真实无人机。它可以接近“运维决策平台”，但不能变成“飞控执行平台”。

# v0.9.0 Release Readiness Checklist

This checklist closes the v0.9.0 work-order draft loop. It is intentionally
local, deterministic, offline-only, and advisory-only.

## Scope

v0.9.0 release readiness covers:

- Offline work order draft generation from `maintenance_recommendations.json`.
- Offline work order draft validation.
- Optional work order draft and validation sections in `ops_report.md`.
- Audit records for work order drafting and work order validation.
- Existing report, simulation, monitoring, preflight, PDF, PX4 mock, ArduPilot
  mock, and real fixture placeholder regression tests through `pytest`.

It does not add real drone control, real maintenance-system integration, real
dispatch, real simulator launch, real hardware connection, fleet analytics, or
web dashboard behavior.

## Required Local Checks

Run the full test suite:

```bash
pytest
```

Generate a temporary report directory:

```bash
python -m apps.cli.main run-mvp \
  --log data/sample_logs/example_flight.csv \
  --asset data/sample_assets/uav_001.json \
  --out <tmp-report-dir>
```

Generate local work order drafts:

```bash
python -m apps.cli.main generate-work-orders \
  --maintenance <tmp-report-dir>/maintenance_recommendations.json \
  --asset data/sample_assets/uav_001.json \
  --out <tmp-report-dir>
```

Confirm these files exist:

- `<tmp-report-dir>/work_order_drafts.json`
- `<tmp-report-dir>/work_order_drafts.md`
- `<tmp-report-dir>/audit/work-order-drafting-*.json`

Validate the local work order drafts:

```bash
python -m apps.cli.main validate-work-orders \
  --drafts <tmp-report-dir>/work_order_drafts.json \
  --out <tmp-report-dir>
```

Confirm these files exist:

- `<tmp-report-dir>/work_order_validation.json`
- `<tmp-report-dir>/audit/work-order-validation-*.json`

Confirm `work_order_validation.json` includes:

- `status=passed`
- `counts.validated_drafts`
- `counts.evidence_refs`
- `created_at=1970-01-01T00:00:00Z`

Regenerate the Markdown report with work order sections:

```bash
python -m apps.cli.main generate-report \
  --summary <tmp-report-dir>/flight_summary.json \
  --anomalies <tmp-report-dir>/anomalies.json \
  --diagnosis <tmp-report-dir>/diagnosis.json \
  --maintenance <tmp-report-dir>/maintenance_recommendations.json \
  --work-orders <tmp-report-dir>/work_order_drafts.json \
  --work-order-validation <tmp-report-dir>/work_order_validation.json \
  --out <tmp-report-dir>/ops_report.md
```

Confirm `ops_report.md` includes:

- `## 7.9 工单草稿`
- `## 7.10 工单验证`
- `不会自动派单`

Run the focused work order tests when reviewing v0.9.0 readiness:

```bash
pytest tests/unit/test_work_order_drafts.py \
  tests/unit/test_work_order_validation.py \
  tests/integration/test_work_order_cli.py \
  tests/integration/test_work_order_validation_cli.py \
  tests/unit/test_audit_and_report.py \
  tests/integration/test_cli_workflow.py
```

## GitHub Actions Gate

Before publishing v0.9.0, push the final v0.9.0 branch to GitHub and confirm
the latest CI run is successful for:

- Python 3.11
- Python 3.12

The existing CI already runs `pytest`. Avoid expanding CI for v0.9.0 unless a
future PR introduces release-critical behavior that is not covered by the
current suite.

## Safety Boundary

v0.9.0 remains an offline advisory work-order draft release.

Do not add:

- Real drone connection.
- MAVLink command execution.
- Arm/disarm.
- Takeoff, landing, or RTL.
- Mission execution.
- Motor start.
- Firmware upload.
- Flight-controller parameter writing.
- PX4, ArduPilot, Gazebo, SITL, or external simulator startup or connection.
- Real CMMS, Jira, Feishu, WeCom, or other maintenance-system API calls.
- Automatic dispatch.
- Automatic maintenance execution.
- Real, sensitive, or binary flight logs.

Work order outputs remain `DRAFT`. They require qualified human review and do
not authorize real maintenance activity.
